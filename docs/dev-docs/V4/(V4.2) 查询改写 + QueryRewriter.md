# 查询改写 + QueryRewriter

> [!note|center] 为什么需要查询改写
> V4.1 直接把用户的原始输入丢进 RAG Pipeline 做检索。这在用户使用规范的技术术语时效果很好，但实际使用中用户经常用口语化表达——"那玩意儿为啥这么快""Java里那个Map咋用"——这些表达在向量空间中很难匹配到知识库里的专业表述。
>
> 查询改写就是用轻量 LLM 调用把口语转成检索友好的关键词，提升 RAG 召回率。

## 问题场景

| 用户输入（口语） | 知识库中的文本 | 直接检索效果 |
|------|------|------|
| "InnoDB那玩意儿为啥这么快啊" | "InnoDB 存储引擎性能分析：B+Tree 索引..." | ❌ 差——"那玩意儿""为啥""这么快"都是噪音 |
| "Java里HashMap怎么用的" | "HashMap 使用方法与底层实现" | ⚠️ 一般——"里""怎么用"干扰匹配 |
| "docker那个启动命令是啥" | "Docker 常用命令：docker run..." | ⚠️ 一般——"那个""是啥"冗余 |

改写后：

| 改写结果 | 检索效果 |
|------|------|
| "InnoDB 存储引擎 高性能 原理" | ✅ 精准命中 |
| "Java HashMap 使用方法 底层实现" | ✅ 精准命中 |
| "docker run 启动容器 常用参数" | ✅ 精准命中 |

## 实现方案

### 独立 Rewriter Prompt

查询改写使用一个独立的轻量 Prompt，和 Agent System Prompt 分离：

```markdown title="query-rewriter-prompt.md"
你是一个查询改写助手。你的任务是把用户的口语化问题
改写为更适合在知识库中检索的关键词表达。

## 改写规则
1. 去掉口语化表达
2. 提取核心概念
3. 扩展同义词
4. 保持简洁：输出纯粹的关键词/短句
5. 如果原始问题已经很适合检索：直接输出原问题

只输出改写后的检索文本，不要任何解释。
```

关键设计：**只输出改写文本**——不需要 JSON、不需要解释。LLM 返回什么就直接当检索 query 用。

### QueryRewriterImpl

```java
@Service
public class QueryRewriterImpl implements IQueryRewriter {

    public String rewrite(String rawQuery) {
        try {
            ChatResponse response = chatModel.chat(
                SystemMessage.from(loadPrompt()),
                UserMessage.from(rawQuery)
            );
            String rewritten = response.aiMessage().text().trim();
            return rewritten.isBlank() ? rawQuery : rewritten;
        } catch (Exception e) {
            log.warn("[查询改写] 失败，回退原始查询");
            return rawQuery;  // 容错：改写失败直接用原文
        }
    }
}
```

容错策略：改写失败返回原文，不阻塞主流程。

### 改建 AgentChatImpl

在 V4.1 的链路中插入改写步骤：

```java
// V4.1: search → prompt → generate
// V4.2: rewrite → search → prompt → generate

String searchQuery = queryRewriter.rewrite(userMessage);  // ① 改写
List<SearchResult> results = hybridRetriever.search(searchQuery, 5);  // ② 检索
```

```mermaid
flowchart LR
    A["用户输入"] --> B["① 查询改写<br/>QueryRewriter"]
    B --> C["② RAG 检索<br/>改写后的 query"]
    C --> D["③ Prompt 组装<br/>证据 + 历史"]
    D --> E["④ LLM 生成<br/>+ 来源引用"]
    E --> F["⑤ 记忆追加"]
```

V4.1 的 `retrievalMode` 参数预留了 `FORCE` / `TOOL` 两种模式，查询改写在 `FORCE` 模式下自动生效。后续 Tool Calling 模式下 LLM 自主决定是否需要检索，如果决定检索同样会经过改写步骤。

## 改写失败不影响回答

查询改写的容错设计：

```
改写成功 → 用改写后的 query 检索 → 更高召回率
改写失败 → 用原始 query 检索 → V4.1 级别效果，至少能用
```

这是"锦上添花"的设计——改写是在原有能力上叠加的优化，出问题只影响增量不影响存量。

## 和原项目的关系

原项目 QA-Agent 在 `RagSearchService` 中也有查询改写，用的是 `rewriterModel`（独立的 ChatModel Bean）。我们复用同一个 `ChatModel`（DeepSeek）做改写——因为改写是轻量单轮调用，不需要独立模型。后续如果改写请求量大，可以像原项目一样单独分配一个模型实例。

## 效果验证

测试建议：
1. 用口语化表达问一个问题，记录 RAG 命中数
2. 关闭改写（`searchQuery = userMessage`），用同样问题再测一次
3. 对比两轮命中数
