# 文档分块

> [!note|center]+ 文档分块的目的
> 文档分块（Document Chunking）是指***将一个完整文档拆分成若干较小的逻辑片段***，每个片段通常包含一个或多个段落、句子或**语义完整**的内容块。
> 
> 分块的目的主要有：
> - **提升检索效率**：对长文档进行搜索或向量化时，分块可以**缩小检索范围，减少匹配成本**。
> - **优化语义处理**：在 NLP 任务中（如摘要、问答、Embedding 计算），小块文本可以**避免上下文过长**导致模型截断或性能下降。
> - **便于版本管理和更新**：单独修改文档某个块时，不必处理整个文档，提高维护效率。
> - **支持并行处理**：可以将***不同块并行计算 Embedding*** 或进行 NLP 分析，提高系统吞吐量。

## flexmark 依赖

`flexmark` 是一个 **Java Markdown 解析器和渲染器**，可以将 Markdown 文档解析为 AST（抽象语法树），并支持：

- **灵活的扩展机制**：自定义渲染器或节点处理
- **兼容 CommonMark 标准**：保证 Markdown 渲染一致性
- **多种输出格式**：HTML、AST、PDF 等

> [!info] 为什么选 flexmark
> 相比于简单的正则匹配按 `#` 切分，flexmark 能准确识别 Markdown 的**层级结构**——哪些是真正的标题（而不是代码块里的 `# 注释`），哪些是代码块、引用块、列表。用 AST 做分块不容易出现边界错误，而且每个节点自带类型信息，方便我们针对不同类型的节点做不同处理。

flexmark 将 Markdown 文本解析后的 AST 节点类型包括：

| 节点类型 | 说明 | 我们的处理方式 |
|----------|------|----------------|
| `Heading` | H1-H6 标题 | 作为分块边界，触发新 chunk |
| `Paragraph` | 普通段落 | 追加到当前 chunk 内容 |
| `FencedCodeBlock` | 围栏代码块 | 追加到当前 chunk 内容 |
| `BulletList` / `OrderedList` | 列表 | 追加到当前 chunk 内容 |
| `ThematicBreak` | 分割线 `---` | 追加到当前 chunk 内容 |
| `BlockQuote` | 引用块 | 追加到当前 chunk 内容 |

## AST 遍历分块逻辑

具体来说，通过 flexmark 提供的 `Parser` 解析器，将读取到的文本内容解析为 AST 抽象语法树——一种将原始文本转换为树形数据结构的方式，树上的每个节点对应 Markdown 中的一个结构单元（标题、段落、代码块等）。

```java title:"Parser 解析 md → AST"
public List<DocumentChunkEntity> chunk(SourceDocumentEntity document) {  
    log.info(  
            "[分块] 开始处理: documentId={}, contentLength={}",  
            document.getId(),  
            document.getRawContent().length()  
    );  

    String rawContent = document.getRawContent();  
    if (rawContent == null || rawContent.isBlank()) {  
        log.warn("[分块] 文档内容为空，返回空列表");  
        return Collections.emptyList();  
    }  

    // Parser.parse() 将 Markdown 文本解析为 AST 文档树
    Document ast = PARSER.parse(rawContent);  
    List<DocumentChunkEntity> chunks = extractChunks(ast, document.getId());  
    log.info("[分块] 处理完成: chunks={}", chunks.size());  
    return chunks;  
}
```

> [!tip] 为什么用 `static final` 的 Parser 实例
> `Parser` 的构造开销不小（内部会初始化所有扩展处理器），但它是**线程安全**的。所以声明为 `private static final` 的类级单例，整个应用共享一个实例，避免每次分块都重新创建。

接着进入核心的分块逻辑 `extractChunks`，具体思路为：

- 遍历 AST 的所有子节点，**按节点类型分两条路径处理**：标题节点触发分块边界切换，非标题节点追加到当前 chunk 的内容缓冲区
- 维护一个**标题路径栈**（`headingStack`），栈底是最外层的 H1，栈顶是当前最深层的标题。当遇到新标题时，根据标题层级弹出栈中 >= 当前层级的标题，然后压入新标题——这样就始终保持了正确的层级路径，比如 `Java基础 > 集合框架 > HashMap`

首先，初始化三个关键的数据结构来支撑遍历过程：

```java 
List<DocumentChunkEntity> chunks = new ArrayList<>();        // ①
LinkedList<String> headingStack = new LinkedList<>();        // ②
StringBuilder currentContent = new StringBuilder();          // ③
```

其中：

- **① `chunks`**：最终结果容器，存放所有切分好的 `DocumentChunkEntity`，按遍历顺序追加
- **② `headingStack`**：标题路径栈（FILO），用于动态构建当前 chunk 的 `title_path`。每遇到新标题时都会维护这个栈，确保栈内始终是从根到当前标题的完整路径
- **③ `currentContent`**：当前 chunk 的文本缓冲区，用 `StringBuilder` 而非直接拼字符串是为了避免大量中间对象——节点内容通过 `\n\n` 连接，直到遇到下一个标题时才会 flush 成一个 chunk

> [!note] 标题路径栈的维护逻辑（核心）
> 假设文档结构如下：
> ```
> # A           ← 栈: [A]
> ## B          ← 栈: [A, B]  
> ### C         ← 栈: [A, B, C]
> ## D          ← 栈: [A, D]  ← B 和 C 被弹出
> ```
> 规则：当遍历到标题 `heading` 时，将栈中 `size >= level` 的元素全部弹出，然后压入当前标题文本。这样任何时候栈的内容就是"从根标题到当前标题"的完整路径，直接 `join(" > ")` 就是 `titlePath`。

---

接着，当遍历到**标题节点**时，触发以下动作：

1. 如果 `currentContent` 非空 → 说明上一个 chunk 已经有内容了，将它封装成 `DocumentChunkEntity` 加入 `chunks`，然后清空 `currentContent`
2. 维护 `headingStack`（按上述规则弹出 + 压入）
3. 开始收集新 chunk 的内容

当遍历到**非标题节点**时（段落、代码块、列表等），直接提取节点文本 `node.getChars().toString()`，用 `\n\n` 拼接到 `currentContent` 后面。注意跳过空文本节点——AST 中可能存在无内容的空白节点。

遍历结束后，检查 `currentContent` 是否还有残留内容——如果有，说明最后一个 chunk 还没有 flush，把它也加入结果集。

```java title:"核心分块循环"
for (Node node : ast.getChildren()) {
    if (node instanceof Heading heading) {
        // === 标题节点：触发分块边界 ===
        int level = heading.getLevel();
        
        // 1. Flush 上一个 chunk
        if (!currentContent.isEmpty()) {
            chunkIndex++;
            chunks.add(buildChunk(documentId, chunkIndex, headingStack, currentContent.toString()));
            currentContent.setLength(0);
        }
        
        // 2. 维护标题路径栈
        while (headingStack.size() >= level) {
            headingStack.pollLast();
        }
        headingStack.addLast(heading.getText().toString());
    } else {
        // === 非标题节点：追加内容 ===
        String text = node.getChars().toString().trim();
        if (!text.isEmpty()) {
            if (!currentContent.isEmpty()) {
                currentContent.append("\n\n");
            }
            currentContent.append(text);
        }
    }
}

// 3. 最后一个 chunk
if (!currentContent.isEmpty()) {
    chunkIndex++;
    chunks.add(buildChunk(documentId, chunkIndex, headingStack, currentContent.toString()));
}
```

## 分块结果与 document_chunk 表

每个 chunk 最终会被持久化到 `document_chunk` 表：

```sql
CREATE TABLE IF NOT EXISTS document_chunk (
    id           VARCHAR(36)  PRIMARY KEY COMMENT 'UUID',
    document_id  VARCHAR(36)  NOT NULL COMMENT '所属文档ID，关联 source_document.id',
    chunk_index  INT          NOT NULL COMMENT '块在文档中的序号（从1开始）',
    title_path   VARCHAR(500) COMMENT '标题路径，如 "Java基础 > 集合 > HashMap"',
    content      MEDIUMTEXT   NOT NULL COMMENT '分块文本内容',
    module_tags  VARCHAR(500) COMMENT '模块标签JSON数组，如 ["java","集合"]',
    created_at   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_document_id (document_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='文档分块表';
```

其中 `module_tags` 是从 `headingStack` 中每层标题直接提取的关键词，序列化为 JSON 数组存储。这个字段后续可以用于按主题标签过滤检索范围。`title_path` 则保留了 chunk 在文档中的完整语义位置，检索时可以直接告诉用户"这个答案来自文档的哪个章节"。

## 上传流程整合

V2.1 的上传流程在 V2.2 中扩展为：

```
POST /api/v1/document/upload
  │
  ├─→ 1. 保存原始文档 → source_document 表（V2.1）
  ├─→ 2. flexmark 解析 → AST 分块    （V2.2 新增）
  └─→ 3. 批量存储分块 → document_chunk 表（V2.2 新增）
```

`DocumentServiceImpl.uploadDocument()` 中三步串联，分块失败不会影响文档本身（文档已先落库）。用户也可以通过 `GET /api/v1/document/{id}/chunks` 随时查看某个文档的分块结果。
