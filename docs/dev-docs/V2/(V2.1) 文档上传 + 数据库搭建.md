# 个人文档仓库

> [!note|center] 需求分析 
> 本系统通过数据库存储用户所管理的笔记文档及相关元数据。为***避免直接操作用户本地文件***所带来的安全风险、权限依赖以及数据一致性问题，系统采用**独立的文档仓库机制**进行统一管理。
> 
> 用户需要主动将希望纳入管理范围的笔记文件上传至系统，由系统接收后存储至专属文档仓库，并**建立对应的索引、分类及版本管理信息**。此后，系统对文档的**检索、编辑、同步、知识库构建**等功能均基于仓库中的文件副本进行操作，而不会直接访问或修改用户本地文件。
> 
> 通过构建独立的文档仓库，一方面能够实现文档数据的统一管理与权限控制，另一方面也为后续的全文检索、向量化处理、知识库构建、版本追踪以及备份恢复等能力提供基础支撑。

## 数据库表设计

首先搭建 `source_document` 表，作为系统的**文档注册中心**——每一条记录代表一份已被系统纳入管理的 Markdown 文档，后续的分块、Embedding、Q&A 生成等任务都通过该表的主键 `id` 关联。

```sql 
CREATE TABLE IF NOT EXISTS source_document (  
    id          VARCHAR(36)  PRIMARY KEY COMMENT 'UUID',  
    file_name   VARCHAR(255) NOT NULL COMMENT '上传时的原始文件名',  
    file_type   VARCHAR(20)  NOT NULL DEFAULT 'MARKDOWN' COMMENT '文件类型，当前仅支持 MARKDOWN',  
    raw_content MEDIUMTEXT   NOT NULL COMMENT '文档完整原始文本内容',  
    ref_count   INT          NOT NULL DEFAULT 0 COMMENT '被 QA 集引用次数',  
    is_deleted  TINYINT      NOT NULL DEFAULT 0 COMMENT '软删除标记：0=正常，1=已删除',  
    created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',  
    updated_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'  
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='文档仓库表';
```

其中各字段的含义及作用如下：

| 字段 | 说明 |
|------|------|
| `id` | 文档在系统中的唯一标识（UUID），由 MyBatis-Plus 的 `IdType.ASSIGN_UUID` 在插入时自动生成，后续所有关联表（分块、Q&A 集等）都通过这个 ID 关联 |
| `file_name` | 上传时的原始文件名，仅做展示用，不作为关联键 |
| `file_type` | 文件类型标识，当前默认为 `MARKDOWN`，预留扩展（未来可能支持 PDF 等） |
| `raw_content` | 文档的完整原始文本，保留一份原文方便后续回溯和重新分块 |
| `ref_count` | 引用计数——当该文档被某个 Q&A 生成任务引用时 +1，用于展示"这个文档被用了几次" |
| `is_deleted` | 软删除标记，配合 MyBatis-Plus 的 `@TableLogic` 注解，查询时自动过滤 `= 1` 的记录 |
| `created_at` / `updated_at` | 时间戳，MySQL 自动维护 |

> [!question] 为什么有本地文件了还要存 MySQL？
> 本地 Obsidian Vault 和后端数据库处在**两个不同层次**——本地是你的原材料仓库，MySQL 的 `source_document` 是系统运行时的"已注册材料目录"。系统需要 UUID 作为稳定锚点来串联分块表、检索表、任务表，文件系统路径做不到这一点；同时 SQL 能提供排序、筛选、软删除等元数据管理能力，这些都是文件系统不提供的。

## 上传 API 的实现

用户通过 `POST /api/v1/document/upload` 上传 Markdown 文件，系统接收后将其存入 `source_document` 表。该过程可以借助 MyBatis-Plus 的 `BaseMapper<T>` 提供的基础 CRUD 方法快速实现，无需手写 SQL。

```java title:"SourceDocumentDao.java"
// 继承 BaseMapper 后自动获得 insert / selectById / selectList 等方法
public interface SourceDocumentDao extends BaseMapper<SourceDocumentPO> {
}
```

在持久化对象中，通过 `@TableId(type = IdType.ASSIGN_UUID)` 让 MyBatis-Plus 在 insert 时自动生成 UUID 主键：

```java title:"SourceDocumentPO.java"
@Data  
@Builder  
@NoArgsConstructor  
@AllArgsConstructor  
@TableName("source_document")  
public class SourceDocumentPO {  
  
    /** 文档唯一标识（UUID），插入时自动生成 */  
    @TableId(type = IdType.ASSIGN_UUID)  
    private String id;  
  
    /** 上传时的原始文件名 */  
    private String fileName;  
  
    /** 文件类型，默认 MARKDOWN */  
    private String fileType;  
  
    /** 文档完整原始文本内容 */  
    private String rawContent;  
  
    /** 被 QA 集引用次数 */  
    private Integer refCount;  
  
    /** 软删除标记：0=正常，1=已删除（MyBatis-Plus @TableLogic 自动过滤） */  
    @TableLogic(value = "0", delval = "1")  
    private Integer isDeleted;  
  
    /** 创建时间 */  
    private LocalDateTime createdAt;  
  
    /** 最近更新时间 */  
    private LocalDateTime updatedAt;  
}
```

> [!info] `@TableLogic` 的作用
> 标记为逻辑删除字段后，MyBatis-Plus 会自动将 `DELETE` 语句转为 `UPDATE is_deleted = 1`，并且在所有查询中自动追加 `WHERE is_deleted = 0`，上层代码完全无感。

上传完成后，用户可以通过两种方式查询文档：

- `GET /api/v1/document/{id}` —— 根据 UUID 查单个文档详情，这里通过 `@PathVariable("id")` 从 URL 路径中提取参数（注意要显式写明参数名，否则 Java 编译默认不保留参数名，Spring 会报 `IllegalArgumentException`）
- `GET /api/v1/document/list` —— 返回所有未删除的文档列表，内部调用 `selectList(null)` 获取全表记录

> [!warning] 中文文件名处理
> 浏览器在上传中文文件名时会自动做 URL 编码（如 `不同路径.md` → `%E4%B8%8D%E5%90%8C%E8%B7%AF%E5%BE%84.md`），需要在 Controller 层用 `URLDecoder.decode(rawFileName, StandardCharsets.UTF_8)` 还原，否则数据库存的是编码后的乱码。

## DDD 层调用链

```
HTTP Request (POST /upload)
       │
       ▼
DocumentController (trigger)  
  └─ 解析 MultipartFile，提取文件名和文本内容
       │
       ▼
DocumentServiceImpl (domain)  
  └─ 校验文件类型和内容，调用 DocumentRepository 持久化
       │
       ▼
DocumentRepositoryImpl (infrastructure)  
  └─ Entity → PO 转换，调 SourceDocumentDao.insert()
       │
       ▼
MySQL: source_document 表
```

这样每一层职责清晰：Controller 只管 HTTP 参数的提取和响应封装，Service 负责业务校验和流程编排，Repository 负责数据转换和持久化。
